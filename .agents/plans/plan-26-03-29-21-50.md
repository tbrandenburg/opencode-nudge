# Plan: Publish `opencode-auto-continue` to npm

**Date**: 2026-03-29  
**Spec**: Add the minimum required build pipeline and `package.json` fields so `opencode-auto-continue` can be published as an npm package and consumed by OpenCode's Bun-based installer.

---

## Codebase Intelligence

### Project Structure

Single-repo (not a monorepo). One deliverable package lives in `auto-continue-plugin/`. The root contains only a `Makefile` task runner, git hooks in `hooks/`, and supporting dirs (`.opencode/`, `docs/`, `.agents/`). No top-level `package.json`. Package manager is **Bun** throughout (evidenced by `bun.lock`, `bun install`, `bun test`, `bun-types`). Build system is the `Makefile` delegating to `bun` and `bunx tsc`.

### Stack

| Concern | Choice |
|---|---|
| Language | TypeScript 5.x, strict mode, ESNext target |
| Runtime | Bun v1.x (executes `.ts` source directly) |
| Module format | ESM only (`"type": "module"`) |
| Plugin SDK | `@opencode-ai/plugin` 1.3.3 |
| Test framework | `bun:test` (built-in) |

### Conventions

- **Naming**: kebab-case files (`idle-handler.ts`), `PascalCase` exports (`AutoContinuePlugin`), `camelCase` functions, `UPPER_SNAKE_CASE` constants (one exception: `sessionStates` — do not replicate)
- **Module organization**: Four-file flat layout in `src/` — entry point → handler → pure logic → types/constants. Zero nesting.
- **Test patterns**: `<module>.test.ts` co-located in `src/`. `bun:test` with `describe`/`it`. Unit tests mock minimally via inline `mock()` factories. E2E uses `spawnSync` against the real binary — no mocks.
- **Config patterns**: Single env var (`OPENCODE_IDLE_THRESHOLD_MS`) parsed in `types.ts` at module load. No dotenv, no feature flags.
- **Import paths**: `.js` extensions on all local imports despite `.ts` source (required ESM+Bun convention).

### Reusable Abstractions

- `getOrCreateState` / `canContinue` / `recordContinuation` in `throttle.ts` — all throttle logic goes here
- `sessionStates` Map in `types.ts` — single source of state
- `make test` / `make lint` / `make typecheck` — CI-equivalent gates

### Tooling Constraints

- `make lint` = `make typecheck` = `bunx tsc --noEmit` — no style linter
- Pre-push git hook runs `make test && make test-e2e` — must pass before any push
- `"noEmit": true` in current `tsconfig.json` — build tsconfig must be separate
- `dist/` is already in `.gitignore` — do not remove that entry

### Patterns to Avoid

- Do not propagate `as any` to production code
- Do not hardcode absolute paths in source or config files
- Do not use `camelCase` for module-level constants (the `sessionStates` exception)
- Do not add a style linter — none exists, YAGNI

## Research Findings

### Libraries and APIs

**Name**: Bun bundler (CLI)  
**Source**: https://bun.sh/docs/bundler  
**Key APIs / patterns used in this plan**:
- `bun build ./src/index.ts --outdir ./dist --target node --packages external` — transpiles TS to JS, leaves `node_modules` imports untouched (correct for a library)
- Output is ESM `.js` by default with `--target node`
- Does **not** emit `.d.ts` declaration files — TypeScript declarations still require `tsc`
- Bun bundler note: "not intended to replace `tsc` for typechecking or generating type declarations"

**Version constraints**: Bun v1.x. `--packages external` flag ensures `@opencode-ai/plugin` is not bundled into the output (it must remain a peer/runtime dep).

```bash
bun build ./src/index.ts --outdir ./dist --target node --packages external
```

---

**Name**: TypeScript compiler (`tsc`)  
**Source**: /microsoft/typescript  
**Key APIs / patterns used in this plan**:
- A separate `tsconfig.build.json` with `"noEmit": false`, `"outDir": "./dist"`, `"declaration": true`, `"declarationMap": true` handles the emit step
- The existing `tsconfig.json` retains `"noEmit": true` for editor/typecheck use — it is not touched
- `"declarationMap": true` enables "go to source" in editors even from the compiled output

```json
{
  "extends": "./tsconfig.json",
  "compilerOptions": {
    "noEmit": false,
    "outDir": "./dist",
    "declaration": true,
    "declarationMap": true
  },
  "exclude": ["src/**/*.test.ts"]
}
```

---

**Name**: npm package.json `exports`, `files`, `prepublishOnly`  
**Source**: https://docs.npmjs.com/cli/v10/configuring-npm/package-json  
**Key APIs / patterns used in this plan**:
- `"files": ["dist"]` — publish only the compiled output; `node_modules`, `src`, `*.test.*` are excluded automatically
- `"exports"` — modern alternative to `"main"`, enables conditional resolution and encapsulation
- `"prepublishOnly"` lifecycle script — runs before `npm publish`, guarantees `dist/` is fresh
- `node_modules` is always excluded from the tarball by npm regardless of `files`

---

**Name**: `opencode-helicone-session` (reference implementation)  
**Source**: https://raw.githubusercontent.com/H2Shami/opencode-helicone-session/main/package.json  
**Why relevant**: Published OpenCode plugin on npm; demonstrates the exact `package.json` shape needed  
**What to copy or adapt**:

```json
{
  "main": "./dist/index.js",
  "module": "./dist/index.js",
  "types": "./dist/index.d.ts",
  "exports": {
    ".": {
      "import": "./dist/index.js",
      "require": "./dist/index.js",
      "types": "./dist/index.d.ts"
    }
  },
  "files": ["dist"],
  "scripts": {
    "build": "bun build index.ts --outdir dist --target node",
    "prepublishOnly": "bun run build"
  }
}
```

**Adaptation notes**:
- Their `build` script does not emit `.d.ts`. We will improve on this by also running `tsc` for declarations, since our plugin is typed and consumers benefit from it.
- They use `"require"` in exports for compatibility — include it even though the package is ESM-only; OpenCode's Bun installer doesn't need it but it costs nothing.
- Their `keywords` include `"opencode"` — this is the discoverability convention on npm.

### Open Source References

**Project**: opencode-helicone-session — https://github.com/H2Shami/opencode-helicone-session  
**Why relevant**: Canonical reference for a published OpenCode npm plugin  
**What to adapt**: `package.json` shape (exports, files, scripts, keywords, peerDependencies pattern)

### Research Gaps

- Bun's `bun build --packages external` does not generate `.d.ts` files. This is confirmed in Bun docs ("not intended to replace tsc for declarations"). Two-step build (bun for JS, tsc for declarations) is required — this is a known pattern in the ecosystem.
- Whether OpenCode's Bun-based npm installer (`~/.cache/opencode/node_modules/`) actually resolves `exports` vs `main` at runtime: assumed to use standard Node/Bun module resolution. The `main` field is kept as a fallback for safety.

### Step 1: Add Build Pipeline

**Objective**: Produce a compiled `dist/` directory containing `index.js` and `index.d.ts` from the TypeScript source, suitable for npm consumption. This is the hard blocker — without it, nothing else works.

**Owner**: coding subagent. Deliverable: `tsconfig.build.json` created, `Makefile` updated with `build` target, `dist/` produced locally and verified.

**Scope of work**:

- `Makefile` — add `build` target

**Task breakdown**:

1. Create `auto-continue-plugin/tsconfig.build.json`:
   ```json
   {
     "extends": "./tsconfig.json",
     "compilerOptions": {
       "noEmit": false,
       "outDir": "./dist",
       "declaration": true,
       "declarationMap": true
     },
     "exclude": ["src/**/*.test.ts"]
   }
   ```
   - `extends` the existing `tsconfig.json` so all strict settings, module resolution, and target are inherited
   - `noEmit: false` overrides the parent's `true`
   - `declaration: true` emits `.d.ts` files so TypeScript consumers get type information
   - `declarationMap: true` emits `.d.ts.map` files for "go to source" in editors
   - `exclude` drops all test files from the build output — they must not appear in `dist/`

2. Add scripts to `auto-continue-plugin/package.json`:
   ```json
   "build": "bunx tsc -p tsconfig.build.json",
   "prepublishOnly": "bun run build"
   ```
   - `build` uses `bunx tsc` (consistent with `make typecheck` which already uses `bunx tsc --noEmit`)
   - `prepublishOnly` ensures `dist/` is always fresh before any `npm publish`

3. Add `build` target to root `Makefile`:
   ```makefile
   build:
   	cd $(PLUGIN_DIR) && bun run build
   ```
   Insert after the `install` target. Update `.PHONY` line to include `build`.

4. Run `make build` from the repo root and confirm `dist/` contains:

### Step 2: Update `package.json` Fields for npm

**Objective**: Add `exports`, `files`, `types`, updated `main`, and metadata fields so the package is correctly described to npm and to consumers. This makes `npm publish` produce a correct, minimal tarball.

**Owner**: coding subagent. Deliverable: updated `auto-continue-plugin/package.json` verified with `npm pack --dry-run`.

**Dependencies**: Step 1 complete (`dist/` exists with correct contents).

**Scope of work**:
- `auto-continue-plugin/package.json` — multiple field additions/updates

**Task breakdown**:

1. Update `"main"` from `"./src/index.ts"` to `"./dist/index.js"`
2. Add `"types": "./dist/index.d.ts"`
3. Add `"exports"` block:
   ```json
   "exports": {
     ".": {
       "import": "./dist/index.js",
       "require": "./dist/index.js",
       "types": "./dist/index.d.ts"
     }
   }
   ```
4. Add `"files": ["dist"]`
5. Move `@opencode-ai/plugin` from `dependencies` to `peerDependencies`. Rationale: OpenCode installs plugins into its own Bun cache (`~/.cache/opencode/node_modules/`). The host environment (OpenCode itself) provides `@opencode-ai/plugin` — bundling it as a direct dep would create a duplicate with a potentially different version. Use `peerDependencies` with a loose range:
   ```json
   "peerDependencies": {
     "@opencode-ai/plugin": ">=1.3.0"
   }
   ```
   Keep `@opencode-ai/plugin` in `devDependencies` so local development and tests still work.
6. Add discoverability metadata:
   ```json
   "description": "OpenCode plugin that automatically sends a continue prompt when a session goes idle",
   "keywords": ["opencode", "opencode-plugin", "auto-continue", "plugin"],
   "license": "MIT"
   ```
   (License field should match the root `LICENSE` file — verify its SPDX identifier.)

## Open Questions and Assumptions

1. **npm account**: The plan assumes the implementer has an npm account and the package name `opencode-auto-continue` is not already taken. Check https://www.npmjs.com/package/opencode-auto-continue before the first publish. The plan does not include the actual `npm publish` command — it only sets up the package to be ready for publishing.

2. **`moduleResolution: "bundler"` + `tsc` emit**: The existing `tsconfig.json` uses `"moduleResolution": "bundler"`, which is a Bun/Vite-specific setting. When used with `tsc` emit, it may affect import path resolution in the output. The `tsconfig.build.json` inherits this setting. If emitted JS contains bare imports without `.js` extensions, the build tsconfig may need `"moduleResolution": "NodeNext"` and `"module": "NodeNext"` to force correct extensioned imports. This must be verified in Step 1.

3. **`@opencode-ai/plugin` version**: Current dep is `1.3.3`. The `.opencode/` directory uses `1.3.5`. The peer dep range `>=1.3.0` is assumed to be compatible. If there are breaking changes between `1.3.3` and `1.3.5`, the range may need narrowing.

4. **License**: Root `LICENSE` file is present but its SPDX identifier was not read. Step 2 must verify and match it in `package.json`. Assumed MIT based on project context.

5. **E2E test binary path**: `src/e2e.test.ts` hardcodes `/home/tom/.opencode/bin/opencode`. This is a pre-existing issue, not introduced by this plan. It is noted but out of scope — the E2E validation step above does not rely on this test; it validates the package shape through different means.
