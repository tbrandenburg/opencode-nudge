## Research Findings

### Libraries and APIs

**Name**: `@opencode-ai/sdk` (latest on `dev` branch, npm version current as of 2026-03-29)  
**Source**: `/anomalyco/opencode` (context7) + `https://github.com/anomalyco/opencode/blob/dev/packages/sdk/js/src/`  
**Key APIs used in this plan**:

- `createOpencode(options?)` — spawns `opencode serve` as a child process, waits for the "server listening" stdout line, returns `{ client, server }`. Locates the binary via `PATH` (calls `spawn("opencode", ...)` — no absolute path). Inherits `process.env` so `~/.config/opencode/` auth is picked up. Passes extra config via `OPENCODE_CONFIG_CONTENT` env var (JSON-serialised). Returns `server.close()` for teardown.

  ```ts
  // packages/sdk/js/src/server.ts (actual source)
  const proc = spawn(`opencode`, args, {
    signal: options.signal,
    env: {
      ...process.env,
      OPENCODE_CONFIG_CONTENT: JSON.stringify(options.config ?? {}),
    },
  })
  ```

- `createOpencode({ config: { ... } })` — the `config` object is serialised to `OPENCODE_CONFIG_CONTENT`. The running server merges it with the project's `opencode.json`. Crucially, plugin registration in `.opencode/opencode.jsonc` is applied at server startup (the server reads it from the project root), so **no extra config is needed to load the plugin**.

- `client.session.create({ body: { title? } })` — creates a new session. Returns `Session` with `.data.id`.

- `client.session.prompt({ path: { id }, body: { parts, model? } })` — sends a prompt synchronously and waits for the AI response. Returns the assistant message.

- `client.event.subscribe()` — opens an SSE stream. Returns `{ stream }` where `stream` is an async iterable of typed event objects. Each event has `.type` and `.properties`.

- `server.close()` — kills the child process. Must be called in `afterAll` / `finally`.

**Version constraints**: The `server.ts` source uses `spawn("opencode", ...)` — it relies on `opencode` being in `PATH`. This is the correct portable behaviour: no absolute path needed. The binary at `~/.opencode/bin/opencode` must be on `PATH`, which it already is (since the current test calls it directly and it works).

**Critical discovery — `OPENCODE_CONFIG_CONTENT` env var**:  
The SDK server launcher does not expose a `cwd` option. It always inherits the test process's `cwd`. Since the test is run from `PROJECT_ROOT` (via `make test-e2e`), the server will read `.opencode/opencode.jsonc` from there, loading the plugin automatically — no change needed to plugin registration.

**Critical discovery — event stream shape** (from context7 v2 example):
```ts
for await (const event of events.stream) {
  if (event.type === "session.status" && event.properties.status.type === "idle") { ... }
}
```
The `session.idle` event emitted by the plugin system (used in current stdout check: `"type=session.idle publishing"`) is a **bus event** separate from `session.status`. The log line `"type=session.idle publishing"` is an internal log emitted by the OpenCode event bus, not an SSE event type directly. The SSE event for session going idle is `session.status` with `status.type === "idle"`.

However: the plugin's own `"continuation prompt injected"` log line is emitted via `client.app.log()`. These logs **do not appear in the SSE event stream** directly — they appear in the server's stdout (accessible via the server process stdout pipe), not via the HTTP event API.

**This is the key architectural constraint**: the SDK's `createOpencode()` internally reads `proc.stdout` only to detect the "server listening" startup line, then stops processing it. There is no SDK API to retrieve structured log entries emitted by `client.app.log()` from the server process. The SSE `/event` endpoint delivers session/message/tool events, not application log events.

**Consequence for assertion strategy**: The `"continuation prompt injected"` assertion must be replaced with an observable side-effect verifiable via the SSE stream. The most reliable equivalent is watching for a second `session.status` idle event (or a `message.updated` event for the assistant) after the continuation prompt has been injected — i.e., confirming that a second AI response was produced.

**Alternative**: Keep the server's stdout accessible. The SDK does not expose it, but the `server` object returned by `createOpencode()` wraps the child process. The process's `stdout` is accessible via the internal `proc` reference — but that is not a public API surface. This approach is fragile and must not be used.

**Clean solution**: Assert on the second `session.status` idle event. After the plugin injects the continuation prompt:
1. OpenCode receives the async prompt.
2. The AI responds (second assistant message appears → `message.updated` event with role `assistant`).
3. The session goes idle again → second `session.status` event with `status.type === "idle"`.

Two consecutive `session.status idle` events prove the full chain ran. This is a stronger, more stable assertion than log-scraping.

### Research Gaps

- **`OPENCODE_IDLE_THRESHOLD_MS` via SDK config**: The env var is read by the plugin at module load time inside the opencode process. The SDK passes extra env via `OPENCODE_CONFIG_CONTENT` only (not arbitrary env vars). However, the SDK's `createOpencodeServer` does `...process.env` — so any env var set in the test process's `process.env` before calling `createOpencode()` will be inherited by the server process. Setting `process.env.OPENCODE_IDLE_THRESHOLD_MS = "0"` before the call will work. ✅ No gap.

- **Port collision**: `createOpencodeServer` defaults to port `4096`. If another opencode instance is running during the test, the port is in use and the server fails to start. The SDK exposes a `port` option — use a non-default port (e.g. `4097`) or rely on the fact that `make test-e2e` is not run concurrently. Flag in known gotchas.

- **SSE stream termination**: The `client.event.subscribe()` stream is infinite. The test must break out of the `for await` loop explicitly using a condition (found event) or an AbortController. This is standard async iterator usage and does not require further research.

**Code changes**:

```ts
/**
 * E2E test for the auto-continue plugin.
 *
 * What this test validates (all five E2E conditions):
 *
 *   1. Real system running    — createOpencode() spawns a genuine opencode serve
 *                               process; the project .opencode/opencode.jsonc is
 *                               loaded, registering this plugin automatically.
 *   2. Real entry point       — the triggering condition (session going idle after
 *                               the AI responds) happens naturally inside the
 *                               running server; nothing is injected mid-stack.
 *   3. Full code path         — the plugin's handleIdleEvent → promptAsync path
 *                               is exercised; confirmed by a second session.status
 *                               idle event (the AI responded to the continuation).
 *   4. No layer substituted   — every layer runs in the same process; no REST
 *                               call manually simulates any step.
 *   5. Side-effect verified   — the second session.status idle event is the
 *                               external confirmation that the AI processed the
 *                               continuation prompt and went idle again.
 *
 * Requires:
 *   - `opencode` binary in PATH
 *   - A configured AI provider (if absent the test fails loudly — correct and
 *     expected; the environment is not ready for E2E testing)
 *
 * OPENCODE_IDLE_THRESHOLD_MS is set to 0 ms so the continuation fires on the
 * first idle event that follows the initial AI response.
 */

import { describe, it, expect, afterAll } from "bun:test"
import { createOpencode } from "@opencode-ai/sdk"
import { resolve } from "path"

// The server process inherits process.env, so setting this here propagates
// the override into the opencode server and therefore into the plugin.
process.env["OPENCODE_IDLE_THRESHOLD_MS"] = "0"

const PROJECT_ROOT = resolve(import.meta.dir, "../..")
// 90 s: server start + plugin load + AI response + idle event + continuation + second AI response
const E2E_TIMEOUT_MS = 90_000

let opencode: Awaited<ReturnType<typeof createOpencode>>

describe("auto-continue plugin — E2E", () => {
  afterAll(() => {
    opencode?.server.close()
  })

  it(
    "injects a continuation prompt after the AI responds to an initial message",
    async () => {
      // ── 1. Start a real opencode server (picks up .opencode/opencode.jsonc) ──
      // process.cwd() must be PROJECT_ROOT when the test runs (make test-e2e
      // sets cwd correctly; the server inherits it and loads the project config).
      opencode = await createOpencode({
        timeout: 15_000,
      })
      const { client } = opencode

      // ── 2. Create a session ──────────────────────────────────────────────────
      const sessionResp = await client.session.create({ body: {} })
      const sessionID = sessionResp.data!.id

      // ── 3. Open the event stream before sending the prompt ───────────────────
      const events = await client.event.subscribe()

      // ── 4. Send the initial prompt (waits for the first AI response) ─────────
      await client.session.prompt({
        path: { id: sessionID },
        body: {
          parts: [{ type: "text", text: "say the word OK and nothing else" }],
        },
      })

      // ── 5. Wait for the second idle event (proves continuation was injected) ──
      // The sequence is:
      //   a) first AI response → server emits session.status idle
      //   b) plugin fires continuation prompt (async)
      //   c) second AI response → server emits session.status idle  ← assert this
      let idleCount = 0
      const deadline = Date.now() + E2E_TIMEOUT_MS

      for await (const event of events.stream) {
        if (event.type === "session.status") {
          const props = event.properties as { sessionID: string; status: { type: string } }
          if (props.sessionID === sessionID && props.status.type === "idle") {
            idleCount++
            if (idleCount >= 2) break
          }
        }
        if (Date.now() > deadline) break
      }

      expect(
        idleCount,
        [
          "Did not observe a second session.status idle event.",
          "Possible causes:",
          "  • No AI provider configured (set one up before running E2E tests)",
          "  • Plugin was not loaded — check .opencode/opencode.jsonc registration",
          "  • chat.message hook did not fire — lastUserMessage was never recorded",
          "  • throttle blocked the continuation (check hourly cap / cooldown)",
          "  • promptAsync threw an error inside the plugin",
        ].join("\n")
      ).toBeGreaterThanOrEqual(2)
    },
    E2E_TIMEOUT_MS
  )
})
```

**Key design decisions**:
- `process.env["OPENCODE_IDLE_THRESHOLD_MS"] = "0"` at module scope — the SDK's `createOpencodeServer` does `{ ...process.env }` so the value propagates.
- `cwd` is not passed to `createOpencode()` because the SDK does not expose it. The test relies on `make test-e2e` being run from `PROJECT_ROOT`. This is already the case (`Makefile` uses relative paths that assume the repo root). **Document this assumption** in the Makefile comment.
- `afterAll` handles teardown even if the test throws.
- The `for await` loop breaks on `idleCount >= 2` or deadline — no infinite hang.
- Type assertion on `event.properties` is needed because the SDK types the properties as `unknown` or a union; a minimal cast is acceptable here (same pattern as `as any` at mock boundary in unit tests).

**Verification**:
- `bunx tsc --noEmit` passes.
- `make test` (unit tests) still passes — they are unaffected.

**Validation**:
- `make test-e2e` reaches the assertion and passes.
- If `opencode` is not in PATH, `createOpencode()` throws and the test fails loudly (correct behaviour).
- If no AI provider is configured, `client.session.prompt()` throws or the session never goes idle — the deadline elapses and `idleCount < 2`, causing the assertion to fail loudly (correct behaviour).

**Acceptance criteria**:
- `e2e.test.ts` contains no reference to any absolute path.
- `e2e.test.ts` contains no `spawnSync`.
- `bunx tsc --noEmit` exits 0.
- `make test-e2e` passes end-to-end on a machine where `opencode` is in `PATH` and a provider is configured.

**Definition of done**: File committed, type-checks pass, E2E test passes.

**Known gotchas**:
1. **Port collision**: `createOpencode()` defaults to port `4096`. If another opencode instance is running, the server fails to start with "address in use". Mitigation: use `createOpencode({ port: 4097 })` or ensure no other instance runs during the test. Add a comment.
2. **`session.status` vs `session.idle` event type**: The plugin is triggered by the internal bus event `session.idle` (confirmed in `idle-handler.ts` line 19: `if (event.type !== "session.idle") return`). This is a **plugin-internal event type** received by the plugin via the OpenCode event hook system. The SSE stream exposes `session.status` (with `status.type: "idle"`) as the observable external equivalent. Do not confuse these two — the SSE stream will never emit `event.type === "session.idle"`.
3. **First idle fires before the `for await` loop**: The `prompt()` call is synchronous (waits for the AI). By the time `prompt()` returns, the first `session.status idle` event may have already been emitted and missed by the stream subscriber. To avoid this: subscribe to the event stream **before** calling `prompt()`, which is exactly what the plan specifies. SSE is server-push; events emitted while the subscriber is connected are buffered server-side briefly, but events emitted before the subscription opens can be lost. The ordering in the plan (subscribe → prompt) is intentional and critical.
4. **`createOpencode()` does not accept a `cwd` option**: The server inherits the test process's `cwd`. `make test-e2e` must be run from `PROJECT_ROOT`. Document this in the Makefile with a comment.
5. **`opencode.server.close()` in `afterAll`**: If `createOpencode()` throws before assigning `opencode`, the `afterAll` `?.` guard handles the undefined case.
6. **Bun's `afterAll` scope**: `afterAll` at the `describe` level runs after all `it` blocks in that `describe`. This is the correct scope.
